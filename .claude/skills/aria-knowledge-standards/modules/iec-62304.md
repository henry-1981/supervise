# IEC 62304:2006 - Medical Device Software

## Overview

IEC 62304:2006은 의료기기 소프트웨어 수명주기 프로세스에 대한 국제 표준입니다. 소프트웨어 안전 등급 (Safety Class) 분류 및 그에 따른 요구사항을 정의합니다.

## Key Concepts

### Software Safety Classification

의료기기 소프트웨어는 안전 등급 (Safety Class)에 따라 분류됩니다:

**Class A (No injury or damage to health)**
- 소프트웨어 고장으로 인한 부상 없음
- QMS 요구사항 외 추가 요구사항 없음

**Class B (Non-serious injury)**
- 소프트웨어 고장으로 인한 심각하지 않은 부상 가능
- 소프트웨어 요구사항 사양, 상세 설계, 단위/통합 테스트 필요

**Class C (Death or serious injury)**
- 소프트웨어 고장으로 인한 사망 또는 심각한 부상 가능
- Class B 모든 요구사항 + 위험 관리 조치 검증 + 추적성 분석 필요

### Classification Decision Tree

```
Does software failure cause harm?
├─ No → Class A
└─ Yes → Is the harm serious or fatal?
   ├─ Yes → Class C
   └─ No → Class B
```

## Software Life Cycle Processes

### Clause 5: Software Development Process

**5.1 Software Development Planning**
- 개발 계획 수립
- 안전 등급 결정
- 프로세스 선택
- 위험 관리 통합

**5.2 Software Requirements Analysis**
- 소프트웨어 요구사항 정의
- 기능적 및 비기능적 요구사항
- 인터페이스 요구사항
- 안전 요구사항
- 요구사항 검증

**5.3 Software Architecture**
- 소프트웨어 아키텍처 정의
- 컴포넌트 분해
- 인터페이스 정의
- 추적성 확보

**5.4 Software Detailed Design**
- 상세 설계
- 알고리즘 정의
- 데이터 구조 정의
- 인터페이스 명세
- 추적성 확보

**5.5 Software Unit Implementation**
- 소프트웨어 유닛 구현
- 코딩 표준 준수
- 단위 테스트

**5.6 Software Unit Verification**
- 단위 테스트 계획
- 테스트 케이스 작성
- 테스트 결과 기록
- 코드 검토 (Class B/C)

**5.7 Software Integration**
- 통합 전략 수립
- 통합 테스트
- 인터페이스 검증

**5.8 Software Testing**
- 시스템 테스트 계획
- 테스트 케이스 작성
- 테스트 결과 기록
- 회귀 테스트

## Traceability Requirements

### Traceability Matrix

**Required Traceability Links**:
- Software requirements → Architecture
- Architecture → Detailed design
- Detailed design → Source code
- Source code → Unit tests
- Unit tests → Integration tests
- Integration tests → System tests
- System tests → Risk control measures

**Traceability Benefits**:
- Requirement coverage verification
- Impact analysis for changes
- Regulatory compliance demonstration
- Maintenance support

## Safety Class-Specific Requirements

### Class A Requirements

**Documentation**:
- Software development plan
- Software requirements specification
- Software design specification
- Version description

**Testing**:
- Integration testing
- System testing

### Class B Requirements (plus Class A)

**Additional Documentation**:
- Detailed design specification
- Unit test specification
- Integration test specification
- System test specification
- Traceability matrix

**Additional Testing**:
- Unit testing for each software unit
- Code review
- Structural coverage analysis (recommended)

### Class C Requirements (plus Class B)

**Additional Documentation**:
- Risk management file integration
- Traceability analysis report
- Verification of risk control measures

**Additional Testing**:
- Structural coverage analysis (mandatory)
- Modified condition/decision coverage (MC/DC) analysis
- Boundary value analysis
- Stress testing
- Security testing

## Software Tools

### Tool Validation

**When is Tool Validation Required?**
- Tool used in development process
- Tool output used directly in released software
- No automated verification of tool output

**Tool Classification**:
- **Class A Tools**: No validation required
- **Class B Tools**: Qualification required
- **Class C Tools**: Full validation required

**Tool Validation Activities**:
- Tool requirements specification
- Tool design specification
- Tool verification
- Tool validation

### Static Analysis Tools

**Purpose**: Code quality & defect detection

**Common Tools**:
- SonarQube
- Coverity
- Klocwork
- PC-lint

**Usage**:
- Automated code review
- Complexity analysis
- Coding standard compliance
- Security vulnerability scanning

### Dynamic Analysis Tools

**Purpose**: Runtime behavior analysis

**Common Tools**:
- Valgrind (memory leaks)
- GDB (debugging)
- Profilers (performance)

## Common Deficiencies

1. **Safety classification incorrect**: Wrong class assigned
2. **Traceability incomplete**: Missing links between artifacts
3. **Unit testing inadequate**: Insufficient test coverage
4. **Risk management not integrated**: No ISO 14971 linkage
5. **Verification missing**: No design/code review
6. **Configuration management poor**: No version control
7. **Tool validation missing**: Tools not qualified
8. **Testing documentation incomplete**: Missing test reports
9. **Regression testing absent**: No re-testing after changes
10. **Security not addressed**: No security testing

## Best Practices

1. **Follow Safety Class**: Implement all class-specific requirements
2. **Maintain Traceability**: Keep traceability matrix updated
3. **Automate Testing**: Use automated test frameworks
4. **Review Code**: Conduct peer code reviews
5. **Analyze Coverage**: Achieve >80% code coverage
6. **Manage Configuration**: Use version control & CI/CD
7. **Document Decisions**: Keep design rationale documented
8. **Validate Tools**: Qualify all development tools
9. **Integrate Risk Management**: Link to ISO 14971 process
10. **Plan Maintenance**: Establish maintenance procedures
