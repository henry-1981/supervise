## 1. 프로젝트 개요

### 프로젝트명

**ARIA** - AI Regulatory Intelligence Assistant

### 한 줄 요약

의료기기 RA/QA(Regulatory Affairs / Quality Assurance) 전문가를 위한 AI 기반 규제 인텔리전스 어시스턴트. Claude Code 플러그인으로 구현되며, 비개발자 비즈니스 사용자를 대상으로 설계되었다.

### 저장소

- GitHub: `henry-1981/supervise`
- 원래 프로젝트명: "Cowork Supervisor" (Claude Code 멀티 플러그인 오케스트레이션 시스템)
- 리브랜딩 후: **ARIA**

### 핵심 가치

| 항목 | 내용 |
|------|------|
| 대상 사용자 | 의료기기 RA/QA 실무자 (비개발자) |
| 핵심 기능 | 규제 문서 작성 지원, 표준 조회, 인허가 전략 수립 |
| 플랫폼 | Claude Code 플러그인 |
| 설계 철학 | 도메인 전문성 + 사용자 친화성 + 규제 정확성 |

## 2. 배경 및 동기

### 2.1 프로젝트의 기원: Cowork Supervisor

ARIA는 원래 **"Cowork Supervisor"**라는 이름의 Claude Code 플러그인으로 시작되었다. 이 시스템의 목적은 여러 Claude Code 플러그인을 하나의 통합된 파이프라인으로 오케스트레이션하는 것이었다.

**Cowork Supervisor의 원래 아키텍처:**

- **5단계 파이프라인**: Intent Clarification(의도 명확화) -> Capability Discovery(역량 탐색) -> Planning(계획 수립) -> Orchestration(오케스트레이션) -> Aggregation(결과 통합)
- **6개 에이전트**: 각 파이프라인 단계를 담당하는 전문 에이전트 구성
- **범용 설계**: 특정 도메인에 종속되지 않는 범용 멀티 플러그인 오케스트레이터

### 2.2 전환의 계기

Cowork Supervisor를 검토하는 과정에서 몇 가지 근본적인 한계가 식별되었다:

1. **범용성의 한정적 가치**: 범용 오케스트레이터는 어떤 도메인에서도 "그럭저럭" 작동하지만, 어떤 도메인에서도 뛰어나지 않았다
2. **사용자 페르소나의 부재**: 누가 이 도구를 쓸 것인지에 대한 명확한 정의가 없었다
3. **차별화 부족**: 범용 AI 오케스트레이터는 이미 시장에 다수 존재하며, 독자적 가치 제안이 약했다
4. **MoAI-ADK라는 성숙한 벤치마크의 존재**: 범용 오케스트레이션 분야에서 이미 MoAI-ADK가 높은 수준의 구현체를 보유하고 있었다

### 2.3 전면 재설계 결정

점진적 개선(Incremental Improvement)과 전면 재설계(Full Redesign) 사이에서 **전면 재설계**를 선택하였다.

**결정 근거:**

- 기존 5단계 파이프라인은 범용 오케스트레이션에 최적화되어 있어, 도메인 특화로 전환하려면 근본적인 구조 변경이 필요했다
- 에이전트 구조, 워크플로우, 품질 프레임워크 모두를 도메인에 맞게 재설계해야 일관된 사용자 경험을 제공할 수 있었다
- MoAI-ADK를 벤치마크로 삼되, 그 구조를 그대로 복제하는 것이 아니라 핵심 원칙을 추출하여 도메인에 맞게 재해석하는 전략을 채택했다

## 3. MoAI-ADK 벤치마킹 분석

### 3.1 MoAI-ADK 시스템 개요

MoAI-ADK(Agent Development Kit)는 Claude Code를 위한 전략적 오케스트레이터 시스템이다. ARIA 설계의 주요 벤치마크로 활용되었다.

**MoAI-ADK의 핵심 구성 요소:**

| 구성 요소 | 내용 |
|-----------|------|
| 에이전트 구조 | 27개 전문 에이전트 (Manager 8 + Expert 8 + Builder 3 + Team 8) |
| SPEC 워크플로우 | Plan -> Run -> Sync 3단계 개발 워크플로우 |
| TRUST 5 품질 프레임워크 | Tested, Readable, Unified, Secured, Trackable |
| Progressive Disclosure | 3단계 토큰 최적화 (Metadata -> Body -> Bundled) |
| Agent Teams | 병렬 실행을 위한 실험적 팀 모드 |
| Persistent Memory | 세션 간 학습을 위한 영속적 에이전트 메모리 |
| Skills 시스템 | YAML 프론트매터 기반 스킬 정의 및 관리 |

### 3.2 MoAI-ADK에서 차용한 요소

ARIA는 MoAI-ADK의 다음 원칙을 차용하되, 의료기기 RA/QA 도메인에 맞게 재해석하였다:

**차용한 원칙:**

1. **에이전트 기반 아키텍처**: 단일 모놀리식 시스템이 아닌, 전문화된 에이전트의 조합으로 구성
2. **단계별 워크플로우**: 명확한 단계 분리를 통한 토큰 효율성과 작업 품질 확보
3. **품질 프레임워크**: 자동화된 품질 게이트를 통한 일관된 산출물 품질 보장
4. **플러그인 구조**: 코어와 도메인 전문성의 분리를 통한 확장성 확보
5. **MCP 통합**: 외부 도구 및 서비스와의 표준화된 연결

### 3.3 MoAI-ADK와의 차별점

MoAI-ADK를 그대로 채택하지 않고 재설계한 이유와 주요 차이점:

| 측면 | MoAI-ADK | ARIA | 차이 이유 |
|------|----------|------|-----------|
| 대상 사용자 | 개발자 | 비개발자 (RA/QA 실무자) | 사용자 인터페이스와 용어를 비즈니스 맥락에 맞춤 |
| 에이전트 구조 | 평면 27개 | 3계층 16개 | 도메인 특화 에이전트를 위한 계층적 구조 |
| 워크플로우 용어 | Plan-Run-Sync | Brief-Execute-Deliver | 비즈니스 사용자가 직관적으로 이해할 수 있는 용어 |
| 품질 프레임워크 | TRUST 5 (개발 품질) | VALID (규제 준수) | 규제 산업의 검증/추적 요구사항 반영 |
| 도메인 범위 | 범용 소프트웨어 개발 | 의료기기 RA/QA 특화 | 깊은 도메인 전문성 제공 |

## 4. 핵심 설계 결정 및 근거

### 4.1 프로젝트 이름: ARIA

**선정 과정:**

여러 후보 이름을 체계적으로 평가하였다. 평가 기준은 기억하기 쉬운 정도(Memorability), 전문성(Professionalism), 도메인 명확성(Domain Clarity), 발음 용이성, 확장 가능성이었다.

| 후보 | 점수 | 비고 |
|------|------|------|
| **ARIA** (AI Regulatory Intelligence Assistant) | **9/10** | 기억하기 쉽고, 음악적 연상(아리아)으로 우아한 이미지, 규제 도메인 명시적 |
| PRISM | 7/10 | 다각적 분석 이미지는 좋으나 도메인 연결 약함 |
| RegFlow | 6/10 | 규제 도메인 명시적이나 기억하기 어렵고 기술적 느낌 |
| Nexus-RA | 6/10 | RA 도메인 명시적이나 발음 불편 |

**ARIA 선정 근거:**

- 짧고 발음하기 쉬움 (4글자)
- "아리아"라는 보편적 인지도 (음악 용어)로 브랜드 기억력 높음
- AI Regulatory Intelligence Assistant의 약어로 도메인과 기능을 모두 표현
- 국제적으로 통용 가능한 이름
- 향후 도메인 확장 시에도 자연스러움 (ARIA for Pharma, ARIA for Automotive 등)

### 4.2 에이전트 아키텍처: 3계층 16개 에이전트

**MoAI-ADK의 평면 구조(27개) 대비 3계층 구조(16개)를 선택한 이유:**

MoAI-ADK는 개발자를 위한 범용 도구로서 Manager, Expert, Builder, Team 카테고리로 나뉜 27개 에이전트가 모두 동등한 계층에 위치한다. 이 구조는 유연하지만, 비개발자에게는 복잡도가 높다.

ARIA는 3계층으로 단순화하여 사용자가 시스템을 직관적으로 이해할 수 있도록 설계하였다:

**계층 1 - Core (핵심 4개)**

오케스트레이터, 세션 관리, 품질 관리, 컨텍스트 관리 등 시스템 기반 에이전트. 사용자가 직접 인지하지 않는 인프라 계층이다.

**계층 2 - Business (비즈니스 4개)**

문서 작성, 분석/리서치, 프로젝트 관리, 커뮤니케이션 등 범용 비즈니스 기능을 담당하는 에이전트. 도메인에 무관하게 활용 가능하다.

**계층 3 - RA/QA Domain (도메인 전문 8개)**

의료기기 RA/QA에 특화된 에이전트. 규제 전략, 기술 문서, 품질 시스템, 임상 평가, 사후 시장 감시, 위험 관리, 인허가 제출, 표준 적합성 등 RA/QA 실무의 핵심 영역을 각각 담당한다.

**이 구조의 장점:**

- 비개발자도 "핵심 시스템 - 일반 업무 - 전문 업무"의 3단계로 직관적 이해 가능
- 도메인 에이전트만 교체하면 다른 산업(제약, 자동차 등)으로 확장 가능
- 에이전트 수가 16개로 MoAI-ADK의 27개 대비 관리 복잡도 감소

### 4.3 워크플로우: Brief-Execute-Deliver

**MoAI-ADK의 Plan-Run-Sync를 Brief-Execute-Deliver로 재정의한 이유:**

| 단계 | MoAI-ADK 용어 | ARIA 용어 | 변경 이유 |
|------|--------------|-----------|-----------|
| 1단계 | Plan (계획) | Brief (브리핑) | RA/QA 실무에서 "요구사항 브리핑"이 자연스러운 용어 |
| 2단계 | Run (실행) | Execute (실행) | "Run"은 개발 용어, "Execute"는 비즈니스 범용 용어 |
| 3단계 | Sync (동기화) | Deliver (전달) | "동기화"는 기술 용어, "전달"은 산출물 중심 비즈니스 용어 |

**각 단계의 의미:**

- **Brief**: 사용자가 원하는 업무를 자연어로 설명하면, ARIA가 구조화된 업무 명세(Task Brief)로 변환
- **Execute**: Task Brief에 따라 적절한 도메인 에이전트가 업무 수행 (문서 작성, 규제 분석, 위험 평가 등)
- **Deliver**: 산출물을 사용자에게 전달하고, 관련 문서를 Notion 등에 자동 저장

### 4.4 품질 프레임워크: VALID

**MoAI-ADK의 TRUST 5 대신 VALID를 설계한 이유:**

TRUST 5 (Tested, Readable, Unified, Secured, Trackable)는 소프트웨어 개발 품질에 최적화된 프레임워크이다. ARIA의 대상인 규제 산업에서는 다른 품질 기준이 요구된다.

**VALID 프레임워크:**

| 요소 | 전체 이름 | 의미 | TRUST 5 대응 |
|------|-----------|------|-------------|
| **V** | Verified | 검증됨 - 산출물이 규제 요구사항을 충족하는지 검증 | Tested에 대응 |
| **A** | Accurate | 정확함 - 규제 정보와 인용이 최신이고 정확 | Readable에 대응 |
| **L** | Linked | 연결됨 - 모든 주장에 규제 근거 연결 (traceability) | Trackable에 대응 |
| **I** | Inspectable | 검사 가능 - 감사 및 심사 시 추적 가능한 산출물 | Secured에 대응 |
| **D** | Deliverable | 제출 가능 - 규제 기관 제출 형식에 부합 | Unified에 대응 |

**VALID가 규제 산업에 더 적합한 이유:**

- "Verified"는 "Tested"보다 규제 맥락에서 정확한 용어 (V&V: Verification & Validation)
- "Linked"는 규제에서 핵심인 추적성(Traceability)을 명시적으로 표현
- "Inspectable"은 FDA 감사, 공인기관 심사 등의 규제 현실을 반영
- "Deliverable"은 규제 제출(Submission) 준비 상태를 직접적으로 표현

### 4.5 플러그인 구조: aria-core + aria-domain-raqa

**분리 설계의 이유:**

단일 플러그인이 아닌 코어와 도메인 전문성을 분리한 이유:

- **aria-core**: 오케스트레이터, 세션 관리, 품질 프레임워크, 비즈니스 에이전트 등 도메인에 무관한 기반 기능
- **aria-domain-raqa**: 의료기기 RA/QA에 특화된 에이전트, 규제 지식 베이스, 도메인 워크플로우

**분리의 장점:**

1. **확장성**: aria-domain-pharma, aria-domain-automotive 등 도메인 플러그인을 추가하여 다른 산업으로 확장 가능
2. **유지보수성**: 규제 변경 시 도메인 플러그인만 업데이트하면 되므로 코어에 영향 없음
3. **선택적 설치**: 사용자가 필요한 도메인 플러그인만 설치 가능
4. **테스트 용이성**: 코어와 도메인을 독립적으로 테스트 가능

## 5. 대상 도메인 (의료기기 RA/QA) 선정 이유

### 5.1 도메인 선정 배경

의료기기 RA/QA(Regulatory Affairs / Quality Assurance)를 첫 번째 도메인으로 선정한 이유:

1. **높은 전문성 장벽**: RA/QA 업무는 복잡한 규제 체계에 대한 깊은 이해를 요구하며, AI 지원의 가치가 매우 높음
2. **문서 집약적 업무**: RA/QA 업무의 70% 이상이 문서 작성, 검토, 관리이며, AI가 가장 효과적으로 지원할 수 있는 영역
3. **사용자의 도메인 전문성 보유**: 프로젝트 소유자가 의료기기 RA/QA 분야의 실무 경험을 보유하고 있어 도메인 요구사항을 정확히 정의 가능
4. **시장 기회**: 기존 규제 관리 소프트웨어(IQVIA, Veeva 등)는 대기업 중심이며, 중소 의료기기 기업을 위한 AI 기반 도구 부재

### 5.2 대상 규제 체계

ARIA가 지원하는 주요 규제 및 표준 체계:

**미국 FDA 인허가 경로:**

- **510(k)**: 기존 동등 의료기기와의 실질적 동등성(Substantial Equivalence) 입증
- **PMA (Pre-Market Approval)**: Class III 고위험 의료기기를 위한 가장 엄격한 인허가 경로
- **De Novo**: 새로운 유형의 저-중위험 의료기기를 위한 인허가 경로

**국제 표준:**

| 표준 | 범위 | ARIA 지원 내용 |
|------|------|---------------|
| ISO 13485 | 의료기기 품질경영시스템 | QMS 문서 작성, 프로세스 갭 분석 |
| ISO 14971 | 위험관리 | 위험 분석, 위험 평가, 위험 통제 문서화 |
| IEC 62304 | 의료기기 소프트웨어 생명주기 | SW 개발 문서, 분류 지원 |
| IEC 60601 | 의료용 전기기기 안전 및 성능 | 시험 요구사항 식별, 적합성 문서 |

**EU 규제:**

- **EU MDR (Medical Device Regulation)**: 유럽 의료기기 규정. CE 마킹을 위한 적합성 평가 지원

### 5.3 주요 업무 프로세스

ARIA가 지원하는 RA/QA 핵심 업무 프로세스:

1. **Design Controls (설계 관리)**: 설계 입력/출력, 설계 검증/밸리데이션, 설계 이력 파일(DHF) 관리
2. **CAPA (시정 및 예방 조치)**: 부적합 식별, 근본 원인 분석, 시정 조치 계획 수립 및 추적
3. **Post-Market Surveillance (시판 후 감시)**: 불만 처리, 부작용 보고, 시판 후 임상 추적(PMCF)
4. **Clinical Evaluation Report (임상 평가 보고서)**: 임상 데이터 분석, 문헌 검토, 임상 평가 보고서 작성
5. **Regulatory Submission (인허가 제출)**: 510(k) 제출 문서 패키지, 기술 문서(Technical File) 작성

## 6. 기술 스택 및 통합 계획

### 6.1 플랫폼: Claude Code 플러그인

ARIA는 Claude Code 플러그인으로 구현된다. Claude Code의 플러그인 아키텍처를 활용하여:

- **Commands**: `/aria` 명령어를 통한 사용자 인터페이스
- **Agents**: 전문화된 에이전트 정의 (`.claude/agents/`)
- **Skills**: 도메인 지식 및 패턴 정의 (`.claude/skills/`)
- **MCP Servers**: 외부 서비스 통합 (`.mcp.json`)

### 6.2 MCP (Model Context Protocol) 통합

ARIA는 4개의 주요 MCP 서버와 통합된다:

**6.2.1 Notion MCP - 중앙 지식 허브**

- **역할**: 규제 문서, 프로젝트 데이터베이스, 지식 베이스의 중앙 저장소
- **활용 시나리오**:
  - 규제 제출 문서의 버전 관리
  - CAPA 추적 데이터베이스
  - 규제 변경 사항 알림 및 추적
  - 프로젝트별 규제 전략 문서 관리

**6.2.2 Google Workspace MCP - 협업 도구**

- **역할**: 이메일, 문서, 스프레드시트, 캘린더를 통한 팀 협업
- **활용 시나리오**:
  - 규제 기관과의 서신 초안 작성 (이메일)
  - 공유 기술 문서 작성 및 편집 (Google Docs)
  - 규제 요구사항 매트릭스 관리 (Google Sheets)
  - 인허가 일정 및 마일스톤 관리 (Google Calendar)

**6.2.3 Context7 MCP - 규제/표준 문서 조회**

- **역할**: 최신 규제 및 표준 문서의 실시간 조회
- **활용 시나리오**:
  - ISO 13485, ISO 14971 등 표준의 최신 조항 참조
  - FDA 가이던스 문서 조회
  - 규제 요구사항의 정확한 인용 및 참조

**6.2.4 Sequential Thinking MCP - 복잡한 규제 분석**

- **역할**: 복잡한 규제 판단이 필요한 시나리오에서 체계적 사고 지원
- **활용 시나리오**:
  - 의료기기 분류(Class I/II/III) 결정을 위한 체계적 분석
  - 인허가 경로(510(k) vs PMA vs De Novo) 선택을 위한 의사결정
  - 위험 분석에서 복잡한 위해-시퀀스 분석
  - 규제 전략 수립 시 다각적 시나리오 평가

### 6.3 데이터 흐름 아키텍처

```
사용자 입력 (/aria 명령)
    |
    v
[ARIA Core - Orchestrator]
    |
    +---> [Business Agents] ---> Google Workspace MCP (문서/이메일)
    |
    +---> [RA/QA Domain Agents] ---> Context7 MCP (규제 조회)
    |                             |
    |                             +---> Sequential Thinking MCP (복잡 분석)
    |
    +---> [Quality Agent (VALID)] ---> Notion MCP (문서 저장/추적)
```

## 7. 구현 전략

### 7.1 구현 로드맵: 5단계

전체 구현은 5개 단계(Phase)로 나뉘며, 각 단계는 독립적으로 가치를 제공할 수 있도록 설계되었다.

#### Phase 1: Core Framework (코어 프레임워크)

**목표**: 플러그인 골격 구축 및 기본 오케스트레이션 동작 확인

**산출물:**

- Claude Code 플러그인 스켈레톤 (`plugin.json`, 디렉토리 구조)
- 중앙 오케스트레이터 에이전트
- `/aria` 기본 명령어 (Brief-Execute-Deliver 파이프라인 스텁)
- 세션 관리 기본 구조
- 컨텍스트 관리 기본 구조

**완료 기준**: `/aria help`가 동작하고, 기본 오케스트레이션 흐름이 스텁 수준에서 작동

#### Phase 2: Generic Business Agents + VALID Framework

**목표**: 도메인에 무관한 비즈니스 에이전트 구현 및 품질 프레임워크 적용

**산출물:**

- 4개 비즈니스 에이전트 (문서 작성, 분석/리서치, 프로젝트 관리, 커뮤니케이션)
- VALID 품질 프레임워크 구현 (Verified, Accurate, Linked, Inspectable, Deliverable)
- Brief-Execute-Deliver 워크플로우 완전 구현
- 비즈니스 에이전트 간 핸드오프 프로토콜

**완료 기준**: 일반적인 비즈니스 문서 작성 요청을 VALID 품질 기준에 따라 처리 가능

#### Phase 3: RA/QA Specialized Agents + Key Workflows

**목표**: 의료기기 RA/QA 특화 에이전트 구현 및 핵심 워크플로우 활성화

**산출물:**

- 8개 도메인 에이전트 (규제 전략, 기술 문서, 품질 시스템, 임상 평가, 사후 시장 감시, 위험 관리, 인허가 제출, 표준 적합성)
- 핵심 워크플로우: 510(k) 제출 문서 패키지 생성, Design Control 문서화, CAPA 프로세스 지원
- 도메인 지식 베이스 (규제 요구사항, 표준 조항, 템플릿)
- 도메인 에이전트 간 협업 시나리오

**완료 기준**: 510(k) 제출 문서 패키지의 주요 구성 요소를 AI 지원으로 작성 가능

#### Phase 4: MCP Integrations (Notion, Google)

**목표**: 외부 서비스 통합을 통한 실무 환경 연동

**산출물:**

- Notion MCP 통합: 규제 문서 저장, CAPA 데이터베이스, 프로젝트 추적
- Google Workspace MCP 통합: 문서 협업, 이메일 초안, 일정 관리
- Context7 MCP 통합: 규제/표준 실시간 조회
- Sequential Thinking MCP 통합: 복잡한 규제 의사결정 지원

**완료 기준**: 전체 워크플로우가 외부 서비스와 연동되어 End-to-End로 동작

#### Phase 5: Advanced Features

**목표**: 고급 기능을 통한 시스템 성숙도 향상

**산출물:**

- Persistent Memory: 세션 간 프로젝트 컨텍스트 유지
- Agent Teams: 복잡한 규제 업무의 병렬 처리
- Advanced Analytics: 규제 동향 분석, 위험 통계, 프로젝트 대시보드
- 멀티 제품 관리: 여러 의료기기 프로젝트 동시 관리
- 감사 준비 모드: FDA 감사, 공인기관 심사 대비 문서 패키지 자동 생성

**완료 기준**: 복수의 의료기기 프로젝트를 동시에 관리하며, 감사 준비 문서 패키지를 자동 생성 가능

### 7.2 개발 방법론

ARIA 프로젝트 자체의 개발은 Hybrid 모드(TDD for new + DDD for legacy)로 수행된다:

- 신규 코드: TDD (Red-Green-Refactor)
- 기존 코드 수정: DDD (Analyze-Preserve-Improve)
- 목표 테스트 커버리지: 85%

## 8. 향후 확장 계획

### 8.1 도메인 확장

ARIA의 코어/도메인 분리 아키텍처는 다음과 같은 도메인 확장을 지원한다:

| 확장 도메인 | 플러그인 | 대상 사용자 |
|-------------|----------|-------------|
| 제약 (Pharma) | aria-domain-pharma | 의약품 RA/QA 전문가 |
| 체외진단 (IVD) | aria-domain-ivd | IVD 규제 전문가 |
| 자동차 | aria-domain-automotive | IATF 16949 / ISO 26262 전문가 |
| 항공우주 | aria-domain-aerospace | AS9100 / DO-178C 전문가 |

### 8.2 기능 확장

- **AI 규제 동향 모니터링**: FDA, EU, 각국 규제 기관의 가이던스 업데이트 자동 추적
- **규제 전략 시뮬레이션**: 다양한 인허가 경로의 시간/비용/위험 비교 분석
- **팀 협업 강화**: 다수의 RA/QA 전문가가 동시에 ARIA를 활용하는 시나리오
- **규제 교육 모드**: 신입 RA/QA 직원을 위한 인터랙티브 교육 기능

### 8.3 플랫폼 확장

현재 Claude Code 플러그인으로 시작하지만, 향후:

- **웹 애플리케이션**: 독립형 웹 기반 인터페이스
- **API 서비스**: 기존 QMS(품질경영시스템) 소프트웨어와의 API 연동
- **모바일 앱**: 현장 감사 시 모바일 지원
